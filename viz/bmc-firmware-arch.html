/// out-of-band firmware lifecycle

BMC / BIOS 固件升级管线

AIDC 门户「BMC/BIOS」模块:Redfish 读固件版本 → 门户固件库(Longhorn 持久化)→ 厂商分发的 driver 刷写 → 任务状态机推进到 staged → 独立重启生效。BIOS 默认只 stage,重启是强确认的独立操作。

执行通道 Redfish-first 固件库 Longhorn PVC 任务 单实例状态机 厂商 技嘉/Dell/超微/曙光/联想 重启 hostname 强确认

厂商 driver 分发

// integrations/firmware/ · get_driver(server) 按 servers.manufacturer 子串归一化 · 资源路径经 collection Members 发现(不硬编码 Self/Embedded.1)

GenericRedfishDriver

  • 技嘉 / 曙光 / 联想 / 默认(normalize_vendor 含中英文别名)
  • 读版本:Systems.BiosVersion + Managers.FirmwareVersion
  • 刷写:MultipartHttpPushUri 三段体 UpdateParameters + OemParameters + UpdateFile
  • 技嘉实测坑:OemParameters 必带 ImageType:BMC/BIOS(大写枚举),且不能带 @Redfish.OperationApplyTime(AMI 拒收)
  • httpx 流式分块推送,不整包进内存;同版本重刷 BMC 返回 prepare flash area failed(预期拦截)

DellRedfishDriver

  • iDRAC:有 Multipart 走 generic;否则 HttpPushUri + SimpleUpdate
  • iDRAC8 老固件无 push/SimpleUpdate → 明确报「需 racadm」
  • racadm CLI 兜底留接口(本期 stub)

SupermicroDriver

  • 读 inventory / 重启复用 generic(只读无需 license)
  • 刷写显式 DriverUnsupportedError:需 sum CLI / OOB license
  • UI 按钮禁用并显示原因

升级任务状态机

// firmware_tasks.py · 进程内 _RUNNING registry(replicas=1 安全)· Semaphore(2) 限并发刷写 · 全程经 AsyncWriteSessionLocal 短事务提交

状态流转

  • pending → uploading:记 previous_version,流式推镜像
  • → flashing:拿 Redfish task URL,每 5s 轮询 PercentComplete
  • → staged(终态):BIOS 待重启 / BMC 已自重启
  • → failed:超时 1800s / driver 不支持 / Redfish exception

BMC 自重启竞态

  • flashing 中连续 30s 连不上 + component=bmc
  • → 等待重连(≤5min)后比对 Managers 版本判定成败
  • BIOS staged 后,下次 power-cycle + 采集到新版本 → 自动转 completed

崩溃恢复

  • 启动 recover_stale_tasks():uploading/flashing 残留置 failed
  • 不自动重放 —— 固件刷写不可盲目重试,人工确认 BMC 后重发
  • 固件文件 + 任务状态都在 PVC/DB,pod 重启不丢
