// 03 · GITOPS WORKFLOW

通用更新流程:为什么改了 ConfigMap 不生效?

所有服务都由 ArgoCD 从 Git 管理。改配置 = 改 Git,不要直接 kubectl edit(会被 ArgoCD 同步回去)。

四步:Git → ArgoCD → ConfigMap → 重启

关键认知:env 型 ConfigMap 改完 Pod 不会感知,必须重启;volume 挂载型 kubelet 约 1 分钟更新文件,但应用通常仍需重启才重读。

1
改 Git

在 k8s-gitops 仓库改 manifest / values,commit + push。

2
ArgoCD 同步

自动检测漂移并 Sync(或 UI 手动 Sync / argocd app sync)。

3
ConfigMap 生效

新 ConfigMap 已下发,但运行中的 Pod 还拿着旧值。

4
滚动重启

kubectl rollout restart deploy/<x> -n <ns> 让 Pod 重读。

🔐 改机密(密码/token):不走 ConfigMap,改 k8s-gitops/sealed-secrets/seal.sh 重新封 SealedSecret → push → controller 自动解密成 Secret。接管已有 Secret 需加注解 sealedsecrets.bitnami.com/managed=true。
// 04 · CHEATSHEET

高频命令速查

本机无 kubeconfig,统一走 k8snode01。先设别名,后续命令用 k。

# 入口 + 别名(在节点上)
ssh yiwei@192.168.6.131
alias k='sudo /var/lib/rancher/rke2/bin/kubectl --kubeconfig /etc/rancher/rke2/rke2.yaml'

k get pods -A | grep -v Running        # 找异常 Pod
k get applications -A                   # ArgoCD 同步状态
k -n <ns> logs <pod> -f                  # 跟日志
k -n <ns> describe pod <pod>             # 排查启动失败(看 Events)
k rollout restart deploy/<x> -n <ns>    # 改 ConfigMap 后让其生效
k get svc -A | grep LoadBalancer        # 谁占了 MetalLB IP
k get pvc -A ; k get certificate -A     # 存储卷 / 证书
k get sealedsecrets -A                  # 封装的机密
k top nodes ; k top pods -A             # 资源占用
