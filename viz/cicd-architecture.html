CI / CD 流水线架构

一次 git push 到代码上线的全自动闭环,全程内网、不依赖 GitHub。内网 GitLab 一肩三挑:CI 触发 + 源码主仓 + ArgoCD 的 GitOps 真相源;buildah 出 SHA 镜像,ArgoCD 从 GitLab sync 部署,GitHub 仅做异地备份。点下面的标签看每条子流程怎么流动。

为什么是这套架构

// 每个"绕一下"都对应一个真实的坑

病根:GitHub 穿墙慢

构建机直连 github.com HTTPS 实测 ~15–66 KB/s(GFW 限速),一次源码 checkout 要 6+ 分钟还常超时。所以整条链不依赖 GitHub 可达性。

解法:GitLab 当主仓 + CI

集群自建 GitLab 既是 aidc 主仓、又是 CI 触发源。push 到内网 192.168.6.243 即触发流水线,runner 直接 checkout ~1.5 秒,全程不出网。

曾经的 GitHub Actions(已退役)

旧架构用 GitHub 当"墙外扳机"、再回头从 GitLab 镜像拉源码,绕且脆(撞过并发部署失败)。2026-06-15 迁成 GitLab 原生 CI,这一圈没了。

SHA 钉版本(GitOps)

镜像不用 :latest,用 git commit 短 SHA。线上跑什么版本 k8s-gitops 写得清清楚楚,回滚就是 git revert。

自托管 runner 全内网

GitLab、registry、k8s API、k8s-gitops 全在内网。GitLab Runner 放进来,触发/构建/推镜像/部署全程不出网。

GitHub 还留着干嘛

纯异地备份:aidc 源 + k8s-gitops 都一条 git push 双推过去。不触发 CI、不被 ArgoCD 读——内网 GitLab 才是触发源和 GitOps 真相源,GitHub 只是镜外存一份不裸奔。

GitLab(主仓 + CI + GitOps 真相源)

  • gitlab.djch.one,Omnibus CE 单 pod @ k8snode02
  • 同时托管 aidc 源 + k8s-gitops,是 ArgoCD 的 GitOps 真相源
  • 既是 CI 触发源(.gitlab-ci.yml)又是 CI bump 的目标仓
  • Longhorn 3 副本 + 每天 02:00 备份 → nfs01(留 7 份)

GitLab Runner

  • djch-build-131 @ k8snode01,标签 djch-build
  • shell executor,以 yiwei 跑(继承 sudo + SSH key)
  • 内网 checkout 源码 + buildah 推 registry + bump k8s-gitops 推 GitLab

ArgoCD GitOps 源全内网

  • 15 个 app 全从内网 GitLab 读 k8s-gitops,GitHub 仅备份
  • 走 SSH:git@192.168.6.243 / gitlab-ssh.djch.one
  • 用只读/写 deploy key,不出网

回滚一次部署

在 k8s-gitops(内网 GitLab)里 git revert 那次镜像 bump → push → ArgoCD 自动把 pod 滚回上一个 SHA。镜像还在 registry,秒退。

DNS / 入口: HTTPS 走 gitlab.djch.one → .240(Traefik ingress + cert-manager,L7 能过);SSH 走 gitlab-ssh.djch.one → .243(MetalLB LoadBalancer)。SSH 过不了 ingress,所以单独给一个 LB IP——这也是 ArgoCD 读 k8s-gitops 走的地址。
注意: GitLab 是单 pod(非 HA),Omnibus 本身不支持多副本。对"内网 CI 源 + 镜像"这用途,3 副本存储 + 每日备份足够;节点故障靠 pod 重调度恢复(几分钟)。
