CI / CD 流水线

一次 git push 到代码上线的全过程。核心思路:GitHub 触发 + GitLab 内网镜像加速 + buildah 构建 + GitOps(ArgoCD)部署 —— 因为 GitHub 穿墙太慢,源码 checkout 改走集群内的 GitLab 镜像。

GitHub Actions 触发 GitLab 内网镜像 buildah 构建 registry.djch.one ArgoCD GitOps SHA 钉版本

FLOW 流水线全景

// 从开发机 push 到 pod 跑上新版本,6 步
1
开发机 push workstation

在工作站 ~/git-ops/aidc-portal 改完代码,git push origin main。

2
双 push:GitLab 先 → GitHub 后 dual-push

一条命令推两个远端:① GitLab(gitlab-ssh:yiwei/aidc,内网镜像)② GitHub(guoyiwei8899/aidc,触发 CI)。

GitLab 排在前面,保证 GitHub 触发 CI 时,这个 commit 的 SHA 已经在镜像里,runner 一定 checkout 得到。
3
GitHub Actions 触发自托管 runner github actions

push 到 main 命中路径过滤 → 分发给 自托管 runner djch-build-131(跑在 k8snode01 / 192.168.6.131,systemd 常驻)。

runner 在内网:registry、k8s 集群、k8s-gitops 都在内网,构建和部署不出网。
4
拉源码 + buildah 构建镜像 build

① 从 GitLab 内网镜像 clone 源码到 $GITHUB_SHA(gitlab-ssh→192.168.6.243,~1.5 秒)② buildah 构建打 SHA tag 的镜像 aidc-backend:<sha> / aidc-frontend:<sha> → 推 registry.djch.one。

这步不走 GitHub:GitHub HTTPS 被 GFW 限到 ~50KB/s,checkout 要 6 分钟还常超时;GitLab 内网镜像 1.5 秒搞定。
5
改 k8s-gitops 镜像 tag → push gitops

把 k8s-gitops/manifests/aidc 里的镜像 tag 钉到这次的 <sha>,commit + push 回 k8s-gitops 仓(deploy key)。

用 精确 SHA 而不是 :latest:部署确定、可审计、好回滚 —— git 里记的就是当前线上版本。
6
ArgoCD 同步 → pod 上新版 argocd

ArgoCD 盯着 k8s-gitops,发现 tag 变了 → 自动同步 → aidc 命名空间的 pod 滚动到 :<sha>。完成。

回滚 = 在 k8s-gitops 里 git revert 那个 bump,ArgoCD 自动滚回旧 SHA。
一句话: 你只管 git push,剩下全自动 —— GitHub 触发、GitLab 喂源码、buildah 出镜像、ArgoCD 上线。约 2–4 分钟,纯构建时间,不再被穿墙卡。

WHY 为什么是这套架构

// 每个"绕一下"都对应一个真实的坑

病根:GitHub 穿墙慢

构建机直连 github.com HTTPS 实测 ~15–66 KB/s(GFW 限速),一次源码 checkout 要 6+ 分钟还常超时。所有"绕路"都是为了避开这条慢链路。

解法:GitLab 内网镜像

集群里自建的 GitLab 当 aidc 源码镜像,runner 从内网 192.168.6.243 走 SSH 拉,1.5 秒。CI 的 checkout 从此跟 GitHub 无关。

双 push 的顺序有讲究

工作站 origin 同时推两端,GitLab 排第一。否则 GitHub 先触发 CI、runner 去 GitLab 拉时镜像还没更新 → checkout 扑空。

SHA 钉版本(GitOps)

镜像不用 :latest,用 git commit 的 短 SHA。线上跑什么版本,k8s-gitops 里写得明明白白;回滚就是 git revert。

自托管 runner 在内网

registry、k8s API、k8s-gitops 全在内网。runner 放进来,构建/推镜像/部署都不出网,只有"触发"这一下用 GitHub。

GitHub 还留着干嘛

① 触发 CI(GitHub Actions)② 存 k8s-gitops(ArgoCD 从这读)③ 双 push 让它当异地备份。源码两边都有,不裸奔。

SAFE 备份与回滚

// 出事怎么退回去 / 数据怎么保

回滚一次部署

在 k8s-gitops 里 git revert 那次镜像 bump → push → ArgoCD 自动把 pod 滚回上一个 SHA。镜像还在 registry,秒退。

GitLab 每日备份 → nfs01

CronJob 每天跑 gitlab-backup,tar(repo+DB)+ gitlab-secrets.json + gitlab.rb 一起存到 nfs01:/data/k8s-backup/gitlab_backup,留最近 7 份。

源码异地双份

双 push 让源码同时在 GitLab(内网) 和 GitHub(外网)。一边挂了另一边还在,天然异地备份。

存储层冗余

GitLab 三个卷都是 Longhorn 3 副本跨节点。单节点宕机数据不丢,pod 自动重调度恢复。

注意: GitLab 是单 pod(非 HA),Omnibus 本身不支持多副本。对"内网 CI 源 + 镜像"这用途,3 副本存储 + 每日备份足够;节点故障靠 pod 重调度恢复(几分钟)。
