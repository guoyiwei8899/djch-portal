为什么是这套架构

// 每个"绕一下"都对应一个真实的坑

病根:GitHub 穿墙慢

构建机直连 github.com HTTPS 实测 ~15–66 KB/s(GFW 限速),一次源码 checkout 要 6+ 分钟还常超时。所有"绕路"都为避开这条慢链。

解法:GitLab 内网镜像

集群自建 GitLab 当 aidc 源码镜像,runner 从内网 192.168.6.243 走 SSH 拉,1.5 秒。CI 的 checkout 从此跟 GitHub 无关。

双 push 顺序有讲究

工作站 origin 同时推两端,GitLab 排第一。否则 GitHub 先触发 CI、runner 去拉时镜像还没更新 → checkout 扑空。

SHA 钉版本(GitOps)

镜像不用 :latest,用 git commit 短 SHA。线上跑什么版本 k8s-gitops 写得清清楚楚,回滚就是 git revert。

自托管 runner 在内网

registry、k8s API、k8s-gitops 全在内网。runner 放进来,构建/推镜像/部署都不出网,只有"触发"用 GitHub。

GitHub 还留着干嘛

① 触发 CI ② 存 k8s-gitops(ArgoCD 从这读)③ 双 push 让它当异地备份。源码两边都有,不裸奔。

关键组件 · 备份 · 回滚

// 这条流水线靠哪些东西撑着 + 出事怎么退

GitLab(镜像 + 备份)

  • gitlab.djch.one,Omnibus CE 单 pod @ k8snode02
  • 存储 Longhorn 3 副本(节点挂了数据不丢)
  • 每天 02:00 自动备份 → nfs01(repo+DB+secrets,留 7 份)

自托管 Runner

  • djch-build-131 @ k8snode01,标签 self-hosted,djch
  • systemd 常驻,装 /opt(避开 SELinux 拦 home 执行)
  • deploy key:拉 GitLab 源码 + 推 k8s-gitops

回滚一次部署

在 k8s-gitops 里 git revert 那次镜像 bump → push → ArgoCD 自动把 pod 滚回上一个 SHA。镜像还在 registry,秒退。

源码异地双份

双 push 让源码同时在 GitLab(内网) 和 GitHub(外网)。一边挂了另一边还在,天然异地备份。

注意: GitLab 是单 pod(非 HA),Omnibus 本身不支持多副本。对"内网 CI 源 + 镜像"这用途,3 副本存储 + 每日备份足够;节点故障靠 pod 重调度恢复(几分钟)。
