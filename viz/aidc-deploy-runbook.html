1 重新部署

// 只改了 Python/前端代码 → 重建镜像;改了卷/env/replicas → 走 k8s-gitops

A · 后端代码改动(最常见)

# 本地从 aidc-portal/backend 打包 → 构建节点构建推送 → 滚动重启
cd ~/git-ops/aidc-portal
tar -czf /tmp/be.tar.gz -C backend --exclude=__pycache__ --exclude='*.pyc' .
scp /tmp/be.tar.gz yiwei@192.168.6.131:/tmp/
ssh yiwei@192.168.6.131 '
  rm -rf /tmp/be && mkdir /tmp/be && tar -xzf /tmp/be.tar.gz -C /tmp/be
  cd /tmp/be && sudo -n buildah bud --layers -t aidc-backend:latest .
  sudo -n buildah push aidc-backend:latest docker://registry.djch.one/aidc-backend:latest
  KC="sudo -n /usr/local/bin/kubectl --kubeconfig=/etc/rancher/rke2/rke2.yaml"
  $KC -n aidc rollout restart deploy/aidc-backend
  $KC -n aidc rollout status deploy/aidc-backend --timeout=180s
'

前端同理:把 backend 换 frontend、镜像换 aidc-frontend、tar 排除 node_modules .next。

B · pod 规格改动(卷 / env / replicas / 镜像引用)

# 改 k8s-gitops/manifests/aidc/*.yaml → push → 强制 ArgoCD 同步
cd ~/git-ops/k8s-gitops
# 编辑 manifests/aidc/backend.yaml ...
git add . && git commit -m "aidc: ..." && git push
ssh yiwei@192.168.6.131 "sudo -n /usr/local/bin/kubectl --kubeconfig=/etc/rancher/rke2/rke2.yaml \
  -n argocd annotate applications.argoproj.io aidc-manifests \
  argocd.argoproj.io/refresh=hard --overwrite"
⚠️ ArgoCD selfHeal 坑(踩过两次):直接 kubectl apply 改 spec 会被 selfHeal 回滚。必须先 git push 到 k8s-gitops,再 hard refresh 触发同步;否则 ArgoCD 用旧 commit 把你的改动打回去。删 secret 前务必先让 manifest 真去掉对它的引用,否则 pod 卡 ContainerCreating。

2 备份

// 三样:数据库 · 固件库 PVC · secrets。都已自动化,可手动触发

数据库(自动日备)

  • CronJob home-media/pg-dumpall-nfs 每日 02:00(Asia/Shanghai)pg_dumpall 全库 → nfs01,含 aidc 库
# 手动立刻备一次
$KC -n home-media create job --from=cronjob/pg-dumpall-nfs aidc-manual-dump
$KC -n home-media logs job/aidc-manual-dump -f

固件库 PVC(Longhorn)

  • Longhorn 卷 aidc-backend-data 自动备份 → nfs://192.168.6.135:/data/k8s-backup
  • 另有 CronJob longhorn-system/pg-backup 每日 19:00
  • 手动:Longhorn UI → Volume → Create Backup

Secrets / 配置

  • aidc-secrets 以 sealed-secret 存在 k8s-gitops/manifests/aidc/secrets.sealed.yaml —— git 即备份
  • ⚠️ 关键:sealed-secrets controller 的私钥要单独备(kubectl -n kube-system get secret -l sealedsecrets.bitnami.com/sealed-secrets-key -o yaml),丢了所有 sealed secret 都解不开
  • TLS 证书 aidc-djch-one-tls 由 cert-manager 自动签发,无需备

3 恢复

// 顺序:secrets/清单 → DB → PVC → 部署

整栈恢复(新集群 / 灾后)

# 1. sealed-secrets controller 私钥先恢复(否则 sealed secret 解不开)
$KC apply -f <从备份导出的 sealed-secrets-key.yaml>
# 2. ArgoCD 重新指向 k8s-gitops → 自动拉起 namespace/secret/ingress/deploy
$KC -n argocd app sync aidc-manifests   # 或在 ArgoCD UI Sync
# 3. 镜像若 registry 还在,直接拉;否则按「重新部署」重建推送

数据库恢复

# 从 nfs01 取 dump,灌回 PG-HA 主库(写经 pgpool postgres 超管)
gunzip -c /nfs/k8s-backup/pg-dumpall-YYYYMMDD.sql.gz \
  | $KC -n home-media exec -i my-pg-ha-postgresql-ha-postgresql-0 -- \
    psql -U postgres

只恢复 aidc 库:从 dumpall 里截 aidc 段,或单独 pg_dump aidc 再 psql。

固件库 PVC 恢复

  • Longhorn UI → Backup → 选 aidc-backend-data 备份 → Restore 成新卷
  • 建对应 PVC 绑定该卷(或改 aidc-backend-data PVC 指向恢复卷)
  • backend Recreate 重启后即挂载 /data/firmware
💡 日常只是误删/回退:DB 用 dump 单库恢复;固件库用 Longhorn 快照回滚;manifest 用 git revert + ArgoCD sync。整栈灾备才走上面全流程。

4 部署优化建议

// 现状能用但有改进空间,按收益排序

① 镜像按 git SHA 打标签(最高收益)

  • 现状 :latest 无法回滚,且 ArgoCD 感知不到镜像变化(踩过「陈旧镜像不回滚」坑)
  • 改 aidc-backend:<gitsha>,k8s-gitops manifest 写死该 tag → ArgoCD 跟踪 + 一键回滚
  • 配 ArgoCD Image Updater 可自动 bump tag

② 修 build-and-deploy.sh

  • 现脚本硬编 /root/aidc-portal,与实际 checkout 路径不符 → 不可直接用(本会话一直手动 tar+scp+buildah)
  • 改成从仓库目录相对打包、参数化镜像 tag

③ CI 化

  • GitHub/Gitea Actions:push → 构建 SHA 镜像 → 自动 bump k8s-gitops tag → ArgoCD 同步
  • 免手工 tar+scp+buildah,部署可追溯、可回滚

④ 流程纪律(已成文)

  • spec 改动只能走 k8s-gitops git + ArgoCD;kubectl apply 会被 selfHeal 回滚
  • SHA 化后镜像改动也走 git,彻底统一到 GitOps,不再 rollout restart
  • replicas=1 是硬约束(进程内固件状态机 + RWO PVC),backend 不能 HA —— 保留
